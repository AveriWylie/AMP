--------------------------------------------------------------------------------------------
chunk.py
--------------------------------------------------------------------------------------------
A palette is a local lookup table that maps small section-specific indices to global block
state IDs.

The reason it exists is compression. There are about 20,000 possible block states in 1.20.1
representing each block in a section with its full global state ID would require 15+ bits
per block. But a typical chunk section might only contain 5-10 different block types. So
instead the server says:

local index 0 → global state ID 0    (air)
local index 1 → global state ID 9    (grass_block)
local index 2 → global state ID 1724 (oak_log)

And then in the long array every block is stored as just 0, 1, or 2 , 2 bits instead of 15.
The palette is what lets you translate those small local indices back to the actual block
type.

Concretely for a section that's mostly stone with some dirt:

palette = [1, 10]   ← index 0 is stone state, index 1 is dirt state
long array = ...packed bits... where each entry is 0 or 1

To get the block at a position you extract the index from the long array, look it up in
the palette to get the global state ID, then look that up in _state_to_block to get the
name.

So the full resolution chain is:
bit index in long array
        ↓
palette index (0, 1, 2...)
        ↓
global block state ID (0-20000)
        ↓
block name ("stone", "dirt", "oak_log"...)
Each step is a lookup. The palette is the middle one

--------------------------------------------------------------------------------------------

bits_per_entry is how many bits the server uses to represent each block's palette index in
the packed long array.

It's determined by the size of the palette — specifically how many bits you need to
represent the largest index in it. If the palette has 2 entries you only need 1 bit per
block (0 or 1). If it has 16 entries you need 4 bits. If it has 256 entries you need 8
bits.

Concretely:

palette size 1        → bits_per_entry = 0  (single value, no array needed)
palette size 2-16     → bits_per_entry = 4  (minimum, clamped up from 1-3)
palette size 17-32    → bits_per_entry = 5
palette size 33-64    → bits_per_entry = 6
palette size 65-128   → bits_per_entry = 7
...and so on
palette size too large → bits_per_entry = 15+ (direct mode, no palette at all)

So it's essentially the server telling you "I packed 4096 block indices into this
long array, and each index is N bits wide — use that to unpack them correctly."

This is why it drives everything downstream — it determines whether there's a palette at
all, how many longs the array needs, how to extract each index with the mask, and how many
blocks fit in each long. Every other piece of the section parse depends on reading this one
byte correctly first.

--------------------------------------------------------------------------------------------

So:

server builds section
        ↓
counts distinct block types → determines palette size
        ↓
palette size → determines how many bits needed per index
        ↓
server writes bits_per_entry into packet
        ↓
server writes palette into packet
        ↓
server packs 4096 indices into long array using bits_per_entry
        ↓
sends it all to you

So on your end when you receive the payload:

read bits_per_entry
        ↓
now you know how wide each index is in the long array
        ↓
read palette (unless bits_per_entry == 0 or >= 15)
        ↓
now you can translate indices to state IDs
        ↓
read data_length → read that many longs
        ↓
now you can extract every block index using bits_per_entry as the mask width
        ↓
translate each index through palette → state ID → block name

I.e. pallet is a long array, in example u gave stone was at index 0, dirt at 1, the later
indices are 0 or 1, and at each index this is the position, telling u weather each position
is dirt or stone without considering other blocks given to us by the packet described by
bits peer entry determined from pallet size

--------------------------------------------------------------------------------------------
compound / list subtree nbt read payload
--------------------------------------------------------------------------------------------
Only the root has the full NBT blob structure, type byte, name, then value. Named children
inside a compound do have a type byte and name prefix, so they're similar, but subtree
compounds don't have that outer wrapper.

By the time _read_nbt_payload is called for them, the type byte and name have already been
consumed by the parent compound's loop. So _read_nbt_payload just sees the raw compound
content, children until TAG_END, with no wrapping blob structure around it.

tldr ...

Each level of the tree is a compound, a dict where every key is a name string and every
value is whatever that named tag contains. The name is independent of the type, it's just a
label. The type is separate information that tells the parser what kind of value follows the
name in the bytes, and determines what Python object ends up as the value in the dict.

So the tree is really a tree of compounds as internal nodes, where at each level names point
to either another compound (going deeper) or a primitive/array (a leaf). The names don't
tell you anything about the type, you have to read the type byte to know what's coming.
But once parsed, the dict just has name → value pairs where the value's Python type reflects
what was in the bytes.

For the heightmap specifically it's only one level deep, root compound whose children are
all named long arrays, so it's a flat dict of name → tuple of longs.

So with this dict we can get various information about the chunk, for ex world surface index
linearly with the lx + lz*16 standard formula, pulling the right 64 bit signed integer from
the tuple.

--------------------------------------------------------------------------------------------
get block
--------------------------------------------------------------------------------------------
It takes absolute world coordinates and does three lookups chained together. First it
converts Y to a section index with (y + 64) >> 4, shifting right by 4 is dividing by 16,
since each section is 16 blocks tall and the world starts at y=-64. That tells you which of
the 24 sections to look in.

Then it gets local coordinates within that section, lx = x & 0xF, ly = y & 0xF, lz = z & 0xF.
ANDing with 0xF is taking the low 4 bits, equivalent to modulo 16, giving you position
within the section rather than absolute world position.

Then it computes the block index within the section, (ly * 16 + lz) * 16 + lx, linearizing
the 3D local coordinate into a flat index into the long array, same idea as lx + lz * 16
for the heightmap but with Y involved since sections are 3D.

Then it extracts the palette index from the long array using bits_per_entry as the mask
width, blocks_per_long = 64 // bits tells you how many block indices fit per long,
long_index and bit_offset tell you exactly where in the array to look, and the mask
extracts just those bits.

Then palette index → state ID → block name through the two lookups we discussed.

--------------------------------------------------------------------------------------------
pathfinding with A* algorithm
--------------------------------------------------------------------------------------------
The open set is a min-heap keyed by f = g + h. g is the actual cost from start to the
current node. h is the heuristic estimate from current to goal. f is the priority, you
always expand the node that looks cheapest total.

The heuristic is what separates A* from Dijkstra. Dijkstra sets h = 0 so f = g only, pure
cost expansion, guaranteed optimal but explores in all directions equally. A* adds h to
guide the search toward the goal, so it explores far fewer nodes. As long as h never
overestimates the true remaining cost (admissible), A* is still guaranteed optimal.

Manhattan distance is admissible here because the actual path can never be shorter than the
sum of absolute coordinate differences on a grid, you can't cut corners, every step costs 1.

The stale entry check (if g > g_score) is needed because heapq in Python has no decrease-key
operation, when you find a better path to a node you push a new entry rather than updating
the old one, so old entries linger and need to be skipped when popped.

--------------------------------------------------------------------------------------------
Pathfinder weight and input mode
--------------------------------------------------------------------------------------------
The pathfinder currently uses a weighted heuristic driven by input mode. Guided mode uses
w=1.0 for optimal paths since the user is directing precisely and deterministically.
Autonomous mode uses w=1.5 for greedier faster searches.

In practice this distinction collapses somewhat. Guided mode is deterministic by nature so
optimal paths are the right call. Autonomous mode will chain tasks anyway so greedy
per-step pathfinding does not meaningfully change macro behavior since the AI planner is
what drives greed not the pathfinder.

The weight is retained as a technical implementation of weighted A* and is a clear
extension point. A more impactful next step is differentiating the modes at the planner
level: guided executes a single intent and waits, autonomous generates a task chain where
each completed step feeds context back into the next AI call as a true agentic loop. The
pathfinder weight can then be tuned more meaningfully once that planner distinction exists.
